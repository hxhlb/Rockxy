# Traffic Rules

Traffic rules are how you move from passive inspection to active debugging. Once Rockxy can see the request, rules let you stop it, change it, reroute it, replace the response, or block it entirely.

Rockxy currently exposes rule types through dedicated tool windows rather than one single generic rules editor. That keeps each workflow explicit and makes it easier to see what the app truly supports today.

## Entry Points

| Action | How to Access |
|--------|---------------|
| Breakpoint Rules window | **Tools > Breakpoint Rules...** (`Cmd+Shift+B`) |
| Breakpoint Queue window | **Tools > Breakpoint Queue...** |

| Map Local quick-create | Right-click a captured request > **Map Local...** |
| Map Remote window | **Tools > Map Remote...** (`Cmd+Opt+R`) |
| Map Remote quick-create | Right-click a captured request > **Map Remote...** |
| Block List window | **Tools > Block List...** (`Cmd+Opt+[`) |
| Block List quick-create | Right-click a captured request > **Tools > Block List...** |
| Allow List window | **Tools > Allow List...** (`Cmd+Opt+A`) |
| Allow List quick-create | Right-click a captured request > **Tools > Allow List...** |
| Allow List sidebar quick-create | Right-click a domain in the sidebar > **Tools > Create Allow List Rule…** |
| Modify Headers window | **Tools > Modify Headers...** |
| Network Conditions window | **Tools > Network Conditions...** (`Cmd+Opt+N`) |
| Network Conditions quick-create | Right-click a captured request > **Tools > Network Conditions...** |

## How to think about rules

Use the simplest rule that proves your hypothesis:

- use **Breakpoint** when you do not yet know the final request or response you want;
- use **Map Local** when you already have the response you want to return;
- use **Map Remote** when you need to redirect traffic to another environment;
- use **Modify Headers** when the payload is fine but metadata is not;
- use **Block** when you want a request to fail immediately;
- use **Allow List** when you want to restrict capture or analysis to a narrow slice;
- use **Network Conditions** when you are testing resilience rather than content.

## Common workflow

1. Capture a real request first.
2. Right-click the request and choose the tool that matches your goal.
3. Let Rockxy prefill the match pattern from the selected request where supported.
4. Keep the rule narrow at first.
5. Re-send the request or trigger it again.
6. Check the result in the inspector, not just in your client.

## Matching strategy

Most Rockxy traffic modification tools work best when you scope them to:

- host or domain,
- path or path prefix,
- HTTP method,
- or a small regex that maps cleanly to the endpoint you are testing.

<Note>
URL patterns use standard regular expressions. To match a literal dot in a domain name, escape it with a backslash: `api\.example\.com`. To match any path under a domain, use `https://api\.example\.com/.*`.
</Note>
